What Does Accessibility Mean to Us?

UX

UX

Operability of the Site

UX :: Operability of the Site

Site Map

  • All sites must have a site map
  • All pages must include the site map link
  • All links of the website must be included
This mainly pertains to smaller websites. Obviously a large blog site cannot contain every single page of the website. At least all main sections and the child pages should be included.
UX :: Operability of the Site

Navigation

  • All links available from the homepage
  • Menu items remain in same location and order
  • Utilize related links
All links from the homepage would again pertain to a smaller website. At the minimum all main sections of the website would be present from the homepage. Really we're talking header and footer navigation. Related Links don't necessarily have to live in a section entitled "Related Links". It's more about being cognizant of all the pages that tie into each other and crosslinking them whether it be in the content or in a specific area
UX :: Operability of the Site

Search Functionality

  • Sites should have search functionality
  • Should include a label, input field and submit button
  • Spell-checker, stemming and synonym extra features will enhance accessibility
Search is especially important in e-commerce sites. The search is only as good as it's results though. If the returned results aren't actually related to the term/s then it would be an accessiblity fail. Luckily the CMS's we use come with built in functionality that is usually pretty good or there are outside search services that are available.
UX

Content

UX :: Content

Show it

  • Don't hide content at one screen size versus another
  • Limit content that is only viewable on hover or click
Content should be thought out strategically. Keeping in mind how text will look on a smaller device will help pare things down into more digestible and easy to read and understandable content. If it's too wordy for mobile, then it's too wordy in general. We shouldn't be hiding content as it gets to smaller sizes. We run into the second item a lot at LMO. Touch devices don't have 'hover'. So the question should always be asked, what is the reason for not showing this content right away? If there's a definite need for it to be hidden initially, there should be a button prompt letting the user know that they have to take an action to view the rest of the content. lose ability to scan a page when hiding content. not all users are able to hover - whether on a mobile device or not
UX :: Content

Logical order/groupings

  • Does the content sequence make sense?
  • Is the webpage easily scannable?
  • Are corresponding pieces of content near each other?
I don't think LMO fails at this. I think we do good with this item for the most part. Again, it helps to think of this one in a mobile-first format, in my opinion. How is mobile content going to read? Then decide what adjustments, if any, need to be made for larger screens. When I say scannable, I mean are the structural components (header, footer, sidebar) easily differentiated from the main content? And is the main content laid out with a page title followed by corresponding page content with quickly readable headers that give meaning to the content on the page? The last point is kind of self explantory.

Meaningful Links

UX :: Meaningful Links

Context

  • Link text should be explanatory - no Click Here or Read More
  • Text surrounding the link should be in relation to the link itself
In a lot of cases this will fall to the developer to handle. We use a class on content that makes it hidden from view but read by screen readers. We tend to use Learn More or Read More when it comes to blog articles, which makes sense from getting a user to understand there is more content elsewhere, but this is a big accessibility failure. But it's also something that is very easy to correct. Example: Learn More About Topic XYZ, where About XYZ would be in the code but hidden from view.
UX :: Meaningful Links

External

  • Only use external links when necessary
  • External links should give users advanced warning
External links can be jarring to those with disabilities. The user is searching a website, the click on a link, and then they're on a whole different site without warning. If the external link is in text, then it would be helpful to explicity say in the text that the following link/s will open in a new site. If it's a standalone link, there should be an icon that signifies a new window will be open and then in code, the developer can use the access class to say that the link will open in a new window.
