What is Accessibility?

A11Y

A11Y is an industry abbreviation of accessibility. The 11 stands for the 11 letters that were removed from the word.

Definition

  • easy to approach, reach, enter, speak with, or use
  • that can be used, entered, reached, etc

Web Accessibility

refers to the inclusive practice of removing barriers that prevent interaction with, or access to websites, by people with disabilities. When sites are correctly designed, developed and edited, all users have equal access to information and functionality.

Aspects of Accessibility

  • Visual
  • Motor/Mobility
  • Auditory
  • Seizures
  • Cognitive/Intellectual
Visual: low vision, blindness, color deficiency Mobility: holding baby, broken arm, missing limbs, diseases like parkinsons Auditory: deaf/partially deaf, in a noisy setting Seizures: self explanatory Cognitive/Intellectual: learning disabilities, age, concussion

WCAG

Web Content Accessibility Guidelines
  • Technical Standard for those in the web industry
  • Version 2.0, but 2.1 is in the works
  • 12 guidelines in 4 principle areas, including success criteria for testing
  • Documentation to meet guidelines, development techniques and further understanding
Reference links open in a new window
https://www.w3.org/WAI/intro/wcag
This is the technical standard for web content developers, web authors, web evaluation tool developers and anyone else that needs to adhere to web accessibility. Its goal is to provide a single, shared standard that meets the needs of everyone, everywhere. Developed by the Accessibility Guidelines Working Group, part of the World Wide Web Consortium (W3C) Web Accessiblity Initiative (WAI).
WCAG

4 Principles

  • Perceivable
  • Operable
  • Understandable
  • Robust
WGAG :: 4 Principles

Perceivable

  • Provide text alternatives for non-text content
  • Provide captions and other alternatives for multimedia
  • Create content that can be presented in different ways, including by assistive technologies, without losing meaning
  • Make it easier for users to see and hear content
WCAG :: 4 Principles

Operable

  • Make all functionality available from a keyboard
  • Give users enough time to read and use content
  • Do not use content that causes seizures
  • Help users navigate and find content
WCAG :: 4 Principles

Understandable

  • Make text readable and understandable
  • Make content appear and operate in predictable ways
  • Help users avoid and correct mistakes
WCAG :: 4 Principles

Robust

  • Maximize compatibility with current and future user tools
Future proofin
WCAG

Levels

  • Level A
  • Level AA
  • Level AAA
WCAG :: Levels

Level A

Most minimal level. Typically the easiest to implement from all standpoints (business logic, design and development). More of an impact on all users, not specific to one type of disability.

WCAG :: Levels

Level AA

What LMO strives to meet. Requires a little more diligence on everybody, but the benefit impact for all users is greater.

WCAG :: Levels

Level AAA

Most difficult to adhere to. In some cases targets more specific disabilities.

The three levels are how the conformity to the guidelines is to be tested. Using generalities to describe these levels. The full webpage must pass all success criteria tests in order to pass the level. Level AAA is not a generally recommended Level to reach because it's hard for certain types of content, or all content, to adhere to these guidelines. However, there are certain Level AAA items, especially in regards to content markup, that are easy enough to do and should be considered standard practice.

Section 508

Refers to the Federal law that requires all Federal agencies to ensure that any electronic and information technology it develops, procures, maintains or uses is accessible to persons with disabilities.

Refreshed in 2017 to update rules, reorganize standards and harmonize with other guidelines and standards such as the European Commission and WCAG. Compliance to begin in January 2018.

Wading through accessibility documentation is a painstaking task. So far there isn't a revised 508 checklist to follow. For any Federal websites LMO made produce, we'll follow the Section 508 rules as we know them today. However, with Section 508 now also adhering to WCAG 2.0 Level AA, so long as we follow those guidelines - and pass them - the websites we produce will also most likely be 508 compliant as well.

What Does Accessibility Mean to Us?

UX

UX

Operability of the Site

UX :: Operability of the Site

Site Map

  • All sites must have a site map
  • All pages must include the site map link
  • All links of the website must be included
This mainly pertains to smaller websites. Obviously a large blog site cannot contain every single page of the website. At least all main sections and the child pages should be included.
UX :: Operability of the Site

Navigation

  • All links available from the homepage
  • Menu items remain in same location and order
  • Utilize related links
All links from the homepage would again pertain to a smaller website. At the minimum all main sections of the website would be present from the homepage. Really we're talking header and footer navigation. Related Links don't necessarily have to live in a section entitled "Related Links". It's more about being cognizant of all the pages that tie into each other and crosslinking them whether it be in the content or in a specific area
UX :: Operability of the Site

Search Functionality

  • Sites should have search functionality
  • Should include a label, input field and submit button
  • Spell-checker, stemming and synonym extra features will enhance accessibility
Search is especially important in e-commerce sites. The search is only as good as it's results though. If the returned results aren't actually related to the term/s then it would be an accessiblity fail. Luckily the CMS's we use come with built in functionality that is usually pretty good or there are outside search services that are available.
UX

Content

UX :: Content

Show it

  • Don't hide content at one screen size versus another
  • Limit content that is only viewable on hover or click
Content should be thought out strategically. Keeping in mind how text will look on a smaller device will help pare things down into more digestible and easy to read and understandable content. If it's too wordy for mobile, then it's too wordy in general. We shouldn't be hiding content as it gets to smaller sizes. We run into the second item a lot at LMO. Touch devices don't have 'hover'. So the question should always be asked, what is the reason for not showing this content right away? If there's a definite need for it to be hidden initially, there should be a button prompt letting the user know that they have to take an action to view the rest of the content. lose ability to scan a page when hiding content. not all users are able to hover - whether on a mobile device or not
UX :: Content

Logical order/groupings

  • Does the content sequence make sense?
  • Is the webpage easily scannable?
  • Are corresponding pieces of content near each other?
I don't think LMO fails at this. I think we do good with this item for the most part. Again, it helps to think of this one in a mobile-first format, in my opinion. How is mobile content going to read? Then decide what adjustments, if any, need to be made for larger screens. When I say scannable, I mean are the structural components (header, footer, sidebar) easily differentiated from the main content? And is the main content laid out with a page title followed by corresponding page content with quickly readable headers that give meaning to the content on the page? The last point is kind of self explantory.
UX

Meaningful Links

UX :: Meaningful Links

Context

  • Link text should be explanatory - no Click Here or Read More
  • Text surrounding the link should be in relation to the link itself
In a lot of cases this will fall to the developer to handle. We use a class on content that makes it hidden from view but read by screen readers. We tend to use Learn More or Read More when it comes to blog articles, which makes sense from getting a user to understand there is more content elsewhere, but this is a big accessibility failure. But it's also something that is very easy to correct. Example: Learn More About Topic XYZ, where About XYZ would be in the code but hidden from view.
UX :: Meaningful Links

External

  • Only use external links when necessary
  • External links should give users advanced warning
External links can be jarring to those with disabilities. The user is searching a website, the click on a link, and then they're on a whole different site without warning. If the external link is in text, then it would be helpful to explicity say in the text that the following link/s will open in a new site. If it's a standalone link, there should be an icon that signifies a new window will be open and then in code, the developer can use the access class to say that the link will open in a new window.

Design

Design

Color

Design :: Color

Contrast Ratios

  • > 24px & > 19px bold = 4.5:1
  • 24+px & 19+px bold = 3:1
  • Color Palette/Branding
  • Hover/Focus states
All text of the page must be test and passing. This means text placed over image, video or multi-colored backgrounds. When we create branding from a project's outset we should be ensuring that the logo and color palette will meet contrast – especially true with buttons. If the colors don't work they'll need to be modified until they do. And if we are given brand standards from an existing guide and the colors will fail contrast, we need to let the client know that their colors will be adjust until they pass the necessary accessibility color contrast ratios. Both hover and focus states must pass the ratios in addition to the default state.
Design :: Color

Conveying Information

  • Text plus color
  • Color plus patterns or icons
We can't rely on color alone to convey meaning. If using color to represent an action or state of something use it in conjunction with text and icons where applicable. The color and pattern item example would be have a route map that uses color to indicate the different paths. In conjunction to the color, a symbol/pattern such as diamonds, squares and circles would also be used.
Design :: Color

Design

Font Guidelines

  • Avoid small font size: 16 pixels is recommended
  • Limit number of fonts used and ensure they're legible
  • Limit the use of bold, italic and all capitals
  • Avoid blinking or moving text
While there is no set specification on a baseline font-size, the development team recommends a starting size of 16px and nothing below 12px if a smaller size is used. The number of fonts used not only can decrease legibility and cause confusion, it can also have a dramatic impact on overall page weight. Font variants should be used only as necessary. Using all caps in large blocks of text makes reading difficulty - limit the use of all caps. Blinking and moving text can cause distractions. There's also a WCAG guideline that requires any autoplaying animations/video must require pause/stop functionality.
Design

Mobile Touch Areas

  • Tap target size is a minimum of 48px
  • Space between tap areas should be a minimum of 32px

Design

Links

  • Contrast with surrounding text is 3:1
  • Contrast with background
  • Indicator (underline)
  • Hover & Focus States
  • External Link Indicator
Link text must pass the requisite color contrast with the background and also provide a sufficient contrast between the surrounding text. However, because color should not be the way to convey something, an indicator should also be used. The best and easiest indicator is underlining the link text. Hover and focus states must also follow the contrast ratios and provide an indication of change. And as mentioned previously, if a link opens in a new window, there should be something to indicate that.
Design

Focus States

  • Browser Defaults
  • Overrides must pass color contrast
  • Anything navigable requires a focus
    • Links, Buttons, Accordions, Menu Buttons, Modal Windows, Tabs/tab content
All browsers have built-in focus states. This is the blue line that people always seem to question. These focus states are necessary. In most cases, the best thing to do is leave this as a browser default. This is an easily changeable item, but if it is changed to more match a site's color scheme, it still must pass color contrast ratio rules.
Design

Forms

Design :: Forms

Guidelines

  • Show labels (placeholders are not labels)
  • Inputs and placeholder text pass color ratios
  • Indicate specific format (date, phone)
Design :: Forms

Guidelines Continued

  • Indicate required fields
    • Asterisk and text
    • Notation must be in the intro text or the first element above the form (visually and html position)
  • Error vs. Success
    • Color, Text and Icons
    • Text is informative
  • Submit
    • No auto submissions (utilize button)
    • In-page completion message should receive focus
Design :: Forms

Financial, Legal or Testing

  • Ability to review before submission and confirm review
  • Ability to recover deleted information
  • Confirmation before continuation
  • Success Feedback
User must review input is correct before completing an irreversible transaction. A checkbox is also required during a review stage or before form submission can be completed. In regards to deleted information, think of Gmail, you can delete items but they're moved to a Trash folder which then has a Permanently Delete action. A user must be prompted with a message asking if they were sure they want to take that action. Think of a form cancel button. Success feedback would be something like creating a user account. The user should be notified their account was created successfully and then told if they are automatically logged in or if they will need to take further action.
Design

Video

  • YouTube Preferred
  • Auto-playing videos must have pause/stop controls (and muted)
  • Many accessibility rules to follow
    • Controls, Closed Caption, Scripts w/time cues, etc

A11Y Quick Points

  • Content reads logically if CSS is turned off
  • Semantic markup and landmark roles
  • Skip link
  • All buttons have text
  • Page title
  • Focus states
  • Keyboard accessible
  • Image Alt attributes
  • Legible at 200%
  • Forms
  • Validation
Semantic markup is probably one of the most important a11y needs as well as one of the easiest to implement or fix. Landmark role of main with an ID for the skip link. Page title as well as all other meta tags (not a11y required) but need to make sure we're including OG tags and Twitter card tags in addition to page title and description meta information. Button markup needs text inside of it. All sections of the site must be keyboard accessible and should show appropriate focus states.
